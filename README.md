# weather-aware-assistant

## Vibe Report

### Where did the AI's "vibe" drift?
The clearest drift happened around Python version compatibility. The AI defaulted to modern type-hint syntax (`date | None`) when writing `calendar_reader.py`, which is valid in Python 3.10+ but crashed immediately on this machine, which runs Python 3.9.6. The AI had no way of knowing the local Python version unless told explicitly — it assumed a newer environment by default. A second, smaller drift was in `main.py`: it wrote `from src.cli import run` without accounting for how the script would actually be invoked (`python3 src/main.py` from the project root), which caused a `ModuleNotFoundError` on first run.

### When did you have to use the "Builder Hammer"?
I didn't write replacement code myself, but I had to manually run the test suite, read the actual tracebacks, and decide what context to feed back to the AI — that diagnostic step was the human part of the loop. For the Python version issue, I copied the exact `TypeError` from the failed test, told the AI which Python version this machine runs, and asked it to fix `calendar_reader.py` and check the rest of the codebase for the same pattern. For the `ModuleNotFoundError`, I again pasted the literal traceback and asked for a fix that didn't require changing how I invoke the script. In both cases, the fix was correct on the first try, but the diagnosis and direction came from manually running the code rather than trusting the AI's own assumptions about the environment.

### What was your most successful "steering" prompt?
The most effective single prompt was the one used to build `calendar_reader.py`, because it specified the function name, the exact input schema, the exact behavior (filter by today's date), the testing approach, and the constraints all in one message: "Implement src/calendar_reader.py. It should read events from calendar.json (schema: list of events with title, start, end, location) and expose a function get_todays_events() that returns only the events whose start date matches today's date. Follow docs/rules.md constraints (type hints, docstrings, graceful error handling for missing/malformed file). Also add a few sample events to calendar.json so we can test this, and write a corresponding test in tests/ using a temporary/mock file — don't depend on the real calendar.json content in the test." This produced a working module, a populated calendar.json, and 8 passing tests in a single pass, with no follow-up correction needed — because the prompt left no ambiguity about scope, schema, or testing strategy.
